# Протокол ANSR-CB (Anonymous Cross-System Risk Reconciliation, UA↔EE)
## Система «Ластівка» — звіряння важливих ризиків дитини без розкриття особи до підтвердженого збігу

### Принципи (інваріанти, що мають виконуватись на кожному кроці)

1. **Compute-to-data.** Сирі персональні дані дитини НІКОЛИ не залишають національний реєстр. Перетинають кордон лише: (а) криптографічні токени фіксованого розміру, (б) булеві/порогові вердикти, (в) зашифрований під ключ авторизованого отримувача payload — і тільки після підтвердженого збігу.
2. **Hit/no-hit двокроковість (модель Prüm/SIS-Art.36).** Крок звіряння дає лише «є збіг високого ризику / немає». Особа розкривається окремою процедурою, тільки на hit, тільки авторизованому отримувачу.
3. **Поріг important-risk = поріг significant harm.** До крипто-звіряння взагалі не доходить нічого, що не перетнуло локальний калібрований поріг ризику + людський перегляд.
4. **Мінімум за побудовою (GDPR Art. 5(1)(c)), а не за політикою.** Кожен крок випускає рівно стільки, скільки потрібно наступному кроку, і ні біта більше.
5. **Захист від реконструкції.** Будь-яке розкриття кардинальності/прапорців обмежене rate-limit + аудитом + (для агрегатів) DP-шумом — проти атак типу AnonPSI.

### Ролі

- **LRA** — Local Risk Agent (легкий модуль на кожному реєстрі: UA-LRA, EE-LRA). Compute-to-data; DataSHIELD-style «дозволені виходи».
- **PEP/PDP** — точка виконання/прийняття рішення про маршрутизацію (ABAC/OPA), локально на стороні отримувача.
- **Converter (опц.)** — нейтральний blind pseudonym converter (CL-15), якщо немає спільного ключа дитини.
- **Транспорт** — X-Road federation (UA Trembita ↔ EE X-Road): point-to-point, message-level signing, hash-chained timestamps. Поверх — onion/cMix для приховання «хто з ким звіряється».

## Огляд потоку (8 кроків)

```
[0] Локальний скоринг + людський gate     (LRA, дані лишаються)
[1] Авторизація права на звірку            (partially-blind signature)
[2] Узгодження ключів епохи + псевдоніми   (VOPRF / CL-15 converter)
[3] Нечіткий блокінг кандидатів            (CLK / Bloom q-gram)        -> лише блок-токени
[4] Криптозвіряння токенів                 (ECDH-PSI, для малих наборів)
[5] Порогове рішення «достатньо важливо»   (threshold-PSI / PSI-CA + DP)
[6] Commit особи (без розкриття)           (Pedersen commitment)
[7] Reveal + маршрутизація отримувачу       (commit-reveal + labeled-PSI/HE + ABAC + PIR)
```

## КРОК 0 — Локальний скоринг ризику та людський gate

**Хто/що обчислює:** UA-LRA (і симетрично EE-LRA) запускає локальний скоринг (rule-based або Isolation Forest / HS-Trees, sub-MB, CPU-only) над сирим записом дитини. Результат — калібрований бал та band. Якщо `band >= CRITICAL`, кейс ставиться в чергу на **людський перегляд** (human-in-the-loop). Лише після підтвердження людиною дитина потрапляє у локальний «набір важливих ризиків» `S_UA`.

**Техніка:** Compute-to-data + DataSHIELD-style disclosure gate (дозволені виходи); lightweight edge anomaly detector; risk-salience поріг = significant-harm gate.

**По дроту передається:** нічого.

**Чому не витікає зайве:** обчислення цілком локальне; на наступні кроки виходить лише факт членства запису в `S_UA` (band, не бал). Людський gate відсіває хибні автоматичні спрацювання ще до будь-якої комунікації (урок AFST: скор — лише тригер тріажу, не рішення).

```text
on_record_update(child_record):
    score, band = local_risk_model(child_record)   # дані не виходять
    if band >= CRITICAL:
        case = enqueue_human_review(child_record, score, band)
        if human_confirms(case):                    # людина-в-петлі
            id_token = stable_local_id(child_record) # лок. ідентифікатор
            S_UA.add(id_token)                       # тільки членство
            audit_log.append(sign("ADDED_TO_RISK_SET", epoch, hash(id_token)))
```

## КРОК 1 — Авторизація права ініціювати звірку (анонімний токен)

**Хто/що обчислює:** UA-LRA просить національний емітент авторизації **засліплено** підписати запит «маю право надіслати сигнал ризику в епоху E, юрисдикція UA, тип=child-protection». Емітент підписує, **не бачачи** прив'язки до конкретної дитини/сесії. UA-LRA знімає засліплення → отримує unlinkable токен авторизації.

**Техніка:** partially-blind signature (Chaum / RSA-BSS, IETF CFRG). Публічна частина = {юрисдикція, тип, епоха}; засліплена частина = nonce сесії.

**По дроту передається:** засліплений запит → підпис. Жодного ідентифікатора дитини.

**Чому не витікає зайве:** blind signature гарантує, що емітент не може зв'язати виданий токен ні з дитиною, ні з майбутньою сесією звіряння (unconditional anonymity). EE-сторона бачить лише валідний токен «дозволено», а не хто/по кому ініціював. Це анти-spam/контроль доступу, який сам не несе PII.

```text
# UA-LRA
m_public  = {jurisdiction:"UA", purpose:"CP", epoch:E}
m_secret  = random_nonce()
blinded   = blind(m_secret, r=random())
sig_b     = Issuer.partially_blind_sign(m_public, blinded)   # Issuer не бачить m_secret
auth_tok  = {m_public, m_secret, unblind(sig_b, r)}          # unlinkable
```

## КРОК 2 — Узгодження ключів епохи та крос-системні псевдоніми

**Хто/що обчислює:** UA-LRA і EE-LRA встановлюють спільні параметри епохи E. Якщо існує надійний спільний ідентифікатор — пропускаємо до Кроку 3. Якщо ні (UA РНОКПП vs EE isikukood) — кожна сторона переводить локальні ID у **секторні псевдоніми** через VOPRF, або нейтральний **converter (CL-15)** сліпо транслює псевдоніми між базами, не бачачи ні псевдонімів, ні особи.

**Техніка:** VOPRF / oblivious pairwise pseudonyms (eIDAS Restricted Identification); blind pseudonym converter Camenisch-Lehmann (CL-15). Ключі — епохні, ротуються.

**По дроту передається:** epoch-параметри; (за converter-варіанту) сліпо замасковані псевдоніми. НЕ передаються: сирі ID, особа, відповідність псевдонімів.

**Чому не витікає зайве:** VOPRF не дає серверу дізнатись вхід; converter за CL-15 не бачить ні вхідних псевдонімів, ні особи, для якої виконує конверсію (гомоморфізм в експонентах + Camenisch-Shoup). Псевдоніми несвязувані між секторами/системами, доки явно не санкціоновано зведення — тому навіть повторні звіряння не накопичують лінкабельний профіль дитини. Епохний ключ обмежує вікно атаки.

```text
# крос-кордонна нормалізація ідентичності (якщо нема спільного ключа)
k_epoch = derive_epoch_key(E)                  # узгоджений секрет епохи
for id in S_UA:
    psd_UA[id] = VOPRF.eval(k_epoch, id)       # сервер не бачить id
# або:
psd_UA = Converter.blind_translate(masked(S_UA))  # converter не бачить псевд./особу
```

## КРОК 3 — Нечіткий блокінг кандидатів (толерантність до транслітерації)

**Хто/що обчислює:** для кожного важливого ризику LRA будує **CLK** (Bloom-енкодинг q-грам ПІБ+ДН, ~1000 біт) — щоб згрупувати потенційно однакових дітей попри різні написання UA(кирилиця)↔EE(латиниця). Сторони обмінюються лише **блок-ключами** (грубі LSH-бакети), а не самими CLK і не записами.

**Техніка:** PPRL — CLK / Bloom q-gram + LSH-блокінг. Призначення: звузити простір кандидатів перед дорогим криптозвірянням.

**По дроту передається:** мінімальні блок-токени (номери LSH-бакетів). НЕ передаються: повні CLK у відкритому вигляді, ПІБ, ДН, ID.

**Чому не витікає зайве:** блокінг розкриває лише грубий бакет (велика анонімна група), не індивід. Критично: **чисті Bloom/CLK НЕ є криптографічно безпечними** (frequency-криптоаналіз, AnonPSI-подібні атаки). Тому CLK тут — лише фільтр кандидатів, а не гарантія приватності; фактичне звіряння робить PSI на Кроці 4. Це і є шаблон «LSH/CLK для кандидатів + PSI для безпечного фіналу».

```text
for id in S_UA:                  # локально
    clk = bloom_qgram(name(id), dob(id))   # ~1000 біт, fuzzy
    bucket = lsh_blocking_key(clk)         # грубий бакет
    send_bucket(bucket)                    # лише бакет, не CLK
# відповідь: спільні бакети-кандидати C_UA ∩ C_EE (грубо)
```

## КРОК 4 — Криптографічне звіряння токенів (PSI)

**Хто/що обчислює:** на множині кандидатів обидві сторони застосовують **спільний OPRF/ECDH** до псевдонім-токенів важливих ризиків і обчислюють перетин. Для малого набору «важливих ризиків» на крос-кордонному WAN обираємо **ECDH-PSI** (2 раунди, ~32–65 байт/елемент, мінімальний трафік; KKRT швидший лише на великих наборах і важчий по комунікації на WAN).

**Техніка:** ECDH-PSI / OPRF (для unbalanced — labeled/unbalanced PSI на стороні великого реєстру). За потреби — PSI-cardinality, що видає лише кількість збігів.

**По дроту передається:** подвійно замасковані точки кривої (фіксований розмір). НЕ передаються: самі ідентифікатори, незбіжні елементи, реєстр.

**Чому не витікає зайве:** PSI за побудовою розкриває лише перетин (або лише його розмір), і нічого про незбіжних дітей. Подвійне маскування приватними ключами обох сторін унеможливлює dictionary-атаку на токени (на відміну від голих хешів). Незбіжні діти жодної зі сторін не розкриваються взагалі — це і є ядро анонімного звіряння.

```text
# ECDH-PSI (спрощено, 2 раунди)
a = sk_UA; b = sk_EE
X_UA = { H(t)^a for t in psd_UA }          # UA -> EE
X_EE = { H(t)^b for t in psd_EE }          # EE -> UA
Z_UA = { x^a   for x in X_EE }             # EE масковані ще й a
Z_EE = { x^b   for x in X_UA }             # UA масковані ще й b
match = Z_UA ∩ Z_EE                        # лише збіги; решта прихована
```

## КРОК 5 — Порогове рішення «достатньо важливо»

**Хто/що обчислює:** замість того, щоб розкривати збіг одразу, протокол спершу перевіряє, чи спільний серйозний ризик перетинає поріг `t` (= поріг significant harm). Використовуємо **threshold-PSI**: фаза Cardinality Testing вирішує «чи перетин ≥ поріг», і лише тоді розкривається сам збіг. На кардинальність накладається **DP-шум** + **rate-limiting** + аудит запитів.

**Техніка:** threshold-PSI (комунікація залежить від `t`, не від розмірів множин) + PSI-CA з диференційною приватністю проти AnonPSI-реконструкції.

**По дроту передається:** булеве «поріг перетнуто / ні» (і, за потреби, зашумлена кардинальність). НЕ передаються: які саме елементи, повний перетин, деталі дитини.

**Чому не витікає зайве:** якщо спільних серйозних ризиків недостатньо для порога — не розкривається нічого взагалі (крипторівневий risk-salience). Навіть розкриття кардинальності небезпечне при повторних запитах (AnonPSI), тому DP-шум + rate-limit обмежують сукупний витік від серій звірянь; бюджет ε витрачається за композицією (kε) і при нулі звіряння припиняються. DP застосовуємо до агрегату/кардинальності, а не до бінарного рішення по конкретній дитині (де шум зруйнував би сигнал).

```text
card = ThresholdPSI.cardinality_test(match)        # лише «> n−t ?»
card_noised = card + Laplace(1/eps)                 # DP проти реконструкції
if rate_limiter.allow(peer, E) and card_noised >= t:
    proceed_to_reveal = TRUE                        # тільки прапор
else:
    abort()                                         # нічого не розкрито
```

## КРОК 6 — Commit особи (фіксація без розкриття)

**Хто/що обчислює:** перед розкриттям обидві сторони **комітять** (приховано, але зобов'язуюче) ідентичність/деталі збіглих дітей. Commitment hiding не дає прочитати значення, binding не дає підмінити його пізніше.

**Техніка:** Pedersen / hash commitment (hiding + binding). Ключ розкриття — під threshold/multi-party контролем (запобіжник: жодна одна сторона не відкриває сама).

**По дроту передається:** commitment (приховане значення). НЕ передається: сама особа.

**Чому не витікає зайве:** на цьому етапі збіг уже криптографічно підтверджений (Крок 5), але особа все ще прихована. Commit фіксує, що сторони зобов'язані саме цими даними (не можна «дописати» дитину постфактум), але не розкриває їх до виконання умови розкриття. Це прямо реалізує вимогу «розкриття особи — лише після підтвердженого збігу високого ризику».

```text
for id in matched_high_risk:
    c = commit(identity(id), r=random())     # hiding + binding
    send(c)                                  # значення приховане
# ключ розкриття розщеплено: reveal вимагає threshold-схвалення
```

## КРОК 7 — Reveal та маршрутизація авторизованому отримувачу

**Хто/що обчислює:**
1. **ABAC/PDP** (OPA/Rego, локально, ~десятки мкс) обчислює `route = f(тип_ризику, роль_отримувача, юрисдикція, ціль)`.
2. Отримувач (поліція/ССД/лікар/EE-орган) доводить право **selective disclosure** credential («уповноважений на ризик типу X у юрисдикції Y») — без зайвих атрибутів.
3. Лише тоді спрацьовує **reveal** (відкриття commitment з Кроку 6); особа/деталі шифруються під ключ авторизованого отримувача (**labeled-PSI/HE** повертає маршрутну мітку разом зі збігом; **PIR** дозволяє отримувачу витягти запис, не розкриваючи, який саме).
4. Доставка — поверх X-Road federation з підписом і hash-chained timestamp (незмінний аудит). Для гострої загрози життю — **break-glass** з обов'язковим незмінним аудитом і пост-ревʼю.

**Техніка:** commit-reveal (тригер = підтверджений збіг + ABAC-дозвіл); ABAC/XACML (PEP/PDP/PIP/PAP) на OPA; selective disclosure VC (SD-JWT за замовчуванням, BBS+ якщо потрібна несвязуваність); labeled/unbalanced PSI + HE для маршрутної мітки; single-server PIR (FrodoPIR/Piano) для приховання патерну доступу; X-Road message-level signing + batch timestamping; break-glass.

**По дроту передається:** зашифрований під отримувача мінімальний payload (особа + мінімум деталей важливого ризику) + маршрутна мітка + підпис/таймстемп. НЕ передається: будь-що по незбіглих/нижчепорогових дітях, повний реєстр, зайві атрибути отримувача.

**Чому не витікає зайве:** розкриття настає лише після (а) криптографічно підтвердженого збігу понад поріг (Кроки 4–5), (б) ABAC-дозволу та (в) доказу повноважень отримувача — три незалежні умови. Selective disclosure гарантує, що й отримувач не розкриває зайвого про себе. HE/labeled-PSI означають, що тільки авторизований отримувач дешифрує payload; координаційний шар маршрутизує, не бачачи значення. PIR ховає, який саме запис витягнуто. X-Road дає non-repudiation без центрального сховища. Break-glass зберігає терміновість, але кожен виклик логується незмінно.

```text
# на стороні отримувача
if proceed_to_reveal and PDP.evaluate(risk_type, recipient_attrs, env) == PERMIT:
    if Recipient.prove_credential(role=X, jurisdiction=Y):   # selective disclosure
        payload = open_commit(c, r)                          # reveal особи
        ct = HE_encrypt(payload, pubkey_recipient)           # лише отримувач дешифрує
        label = labeled_psi_route(match)                     # кому доставити
        xroad_send(signed(ct, label), timestamp_hashchain)   # аудит, no central store
    else: deny()
else:
    discard()        # особа НЕ розкрита
```

| Крок | По дроту | НЕ передається | Техніка | Чому не витікає зайве |
|---|---|---|---|---|
| 0 | нічого | усе сире | compute-to-data, edge ML, human gate | обчислення локальне; виходить лише band після людини |
| 1 | засліплений запит/підпис | ID дитини, сесія | partially-blind signature | емітент не лінкує токен до дитини |
| 2 | epoch-параметри, сліпі псевдоніми | сирі ID, особа, мапінг | VOPRF / CL-15 converter | converter/сервер не бачить вхід; псевдоніми несвязувані |
| 3 | LSH-бакети | CLK, ПІБ, ДН | CLK/Bloom q-gram + LSH | лише груба група; CLK не як гарантія, а як фільтр |
| 4 | масковані точки кривої | ідентифікатори, незбіги | ECDH-PSI / OPRF | розкриває лише перетин; dictionary-атака неможлива |
| 5 | прапор порога (+DP-кардинальність) | елементи, повний перетин | threshold-PSI / PSI-CA + DP | нижче порога — нічого; DP+rate-limit проти AnonPSI |
| 6 | commitment | сама особа | Pedersen commit (hiding+binding) | особа зафіксована, але прихована до умови |
| 7 | ct під отримувача + мітка + підпис | незбіги, реєстр, зайві атрибути | commit-reveal + ABAC/OPA + SD-VC + labeled-PSI/HE + PIR + X-Road | три незалежні умови; лише отримувач дешифрує; патерн доступу прихований |

## Інваріанти безпеки протоколу

- **Поетапне розкриття-монотонність:** на кожному наступному кроці виходить рівно те, що споживає наступний крок (Art. 5(1)(c) за побудовою).
- **No-hit ⇒ no-data:** якщо немає підтвердженого збігу понад поріг, жоден біт PII не перетинає кордон.
- **Захист від серій:** rate-limit + DP-бюджет ε (композиція kε) + незмінний аудит на кожному вузлі (X-Road message log) — проти реконструкції повторними запитами.
- **Поділ довіри:** ключ reveal розщеплено (threshold); converter сліпий; емітент авторизації не бачить дитину; отримувач доводить лише потрібний атрибут.
- **Людина перед розкриттям:** human gate на Кроці 0 і авторизований людський отримувач на Кроці 7 — скор ніколи не діє автоматично (мітигація AFST/LA/Illinois/Oregon).
- **Етичний запобіжник:** обов'язковий bias-аудит локального скорингу; скор — тригер тріажу, не рішення.
