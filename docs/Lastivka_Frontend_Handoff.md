# Промпт для колеги: довести фронтенд «Ластівки» до повного MVP

> Встав це своєму AI-агенту (Claude Code / Cursor) як технічне завдання. Фронтенд пишемо майже з нуля, але спираємось на наявні гілки як референс.

## 1. Контекст проєкту
**«Ластівка» (Lastivka)** — система захисту прав дитини для українсько-естонського хакатону. Суть: **privacy-preserving крос-реєстрове виявлення ризиків** для дітей. Дані з ~17 українських + 4 естонських реєстрів зіставляються, система виявляє «червоні прапорці» й пріоритезує реагування. **Decision support, не вирок** — фінальне рішення завжди за фахівцем.

- Репо: `github.com/mikamika06/lastivka`, локально `D:\Palitech\hackathon\lastivka`
- **Стек фронтенду:** Next.js 16.2.9 (App Router, Turbopack), React 19, TypeScript, **Tailwind CSS v4** (через `@theme` токени → CSS-змінні).
- **Бекенд** (не чіпати): Python FastAPI (`app/api.py`), детермінований пайплайн `realmodel → emitters → matching → detection → scoring`, рівні доступу 1/2/3, крос-кордон PPRL UA↔EE.
- **Демо працює БЕЗ бекенду** — на детермінованих мок-даних (`frontend/src/lib/mock.ts`, seeded PRNG mulberry32), з fallback на реальний API через `NEXT_PUBLIC_API_BASE`.

## 2. Мета
Зробити **завершений MVP-фронтенд**, який:
1. **повністю описує рішення** (що робить, для кого) — як демо + презентація для журі;
2. має **демо-аутентифікацію та рольовий доступ** (нижче);
3. показує **4 стовпи продукту**: Data integration → Risk detection → Targeted response → Management dashboard;
4. **інтегрує ризики** (твоя частина — детально в §7);
5. виглядає **професійно у стилі Дія** (НЕ «ші-дизайн»).

## 4. Дизайн-система (критично — стиль Дія / Мінцифри)
- **НЕ «ші-дизайн».** Чисто, стримано, по-державному. Орієнтир — продукти Мінцифри / Дія.
- **Палітра:** чорно-біла база, **один синій акцент `#2F7BF5`**. Плоско: рамки 1px, мінімум тіней, без градієнтів-«вау».
- **Шрифт:** e-Ukraine / e-Ukraine Head (self-hosted woff2 у `src/app/fonts/`).
- **Світла + темна теми** (темна — у стилі Дія, через `html.dark` + CSS-змінні; анти-FOUC inline-скрипт; localStorage). Перемикач тем у топбарі.
- **Рівні черги (tiers):** T0 «Сьогодні» (червоний), T1 «Цей тиждень» (бурштиновий), T2 «Спостереження» (нейтральний) — статусні кольори.
- **Діаграми — власні SVG** (HBar, Donut, TrendChart зі зламом/change-point). **Без сторонніх chart-бібліотек.**
- **Текст — інтуїтивний для першого глядача, без жаргону й англіцизмів.** Поганий приклад: «Privacy-preserving матчинг (PPRL)». Добре: «приватне зіставлення — між системами передається лише сигнал збігу, не дані». Пояснюй простими словами.
- **Послідовна нумерація** розділів навігації.

## 5. Ролі та доступ (НАЙВАЖЛИВІШЕ — за `ROLE_ACCESS_MATRIX.md`)
Демо показує **рівні доступу**. Аутентифікація демо-рівня (без бекенду): cookie-сесія, сторінка `/login`, захист маршрутів (Next 16 — через **`proxy.ts`**, наступник `middleware.ts`).

**Дві ролі:**

| | **Працівник громади** (Фахівець ССД, Ф-Г) | **Керівник регіону** (Н-Р / Ф-О) |
|---|---|---|
| Бачить | ПІБ, адреси, телефони, кейси, строки, візити | Агрегати, статистику, кількості, повноту даних, стратегію |
| Скоуп | **ЛИШЕ своя громада** | Уся область, але **знеособлено** |
| Персональні дані | Так | **НІ** (ані ПІБ, ані адрес) |
| Екрани | Черга / Профіль / Кабінет (caseload) | Дашборд агрегатів / аналітика |

**Ключові правила:**
- **Ізоляція громад:** працівник громади А **не бачить** кейсів громади Б. Реалізуй фільтр **у шарі даних** (`lib/api.ts`), а не приховуванням у розмітці. Скоуп брати **із сесії**, не з параметрів виклику (щоб закрити IDOR — крос-громадні `?id=` мають повертати null).
- **Для регіонала** ПІБ/УНЗР/дата народження **вирізаються в дата-шарі** (не лише ховаються в JSX).

## 6. Екрани (що показати)
- **Лендінг (`/`)** — опис рішення: проблема, що робить, 4 стовпи, для кого. + сторінка `/about` з детальнішим описом.
- **`/login`** — вибір демо-персони з показом її рівня доступу.
- **Управлінська панель (`/dashboard`)** *[регіонал]* — KPI (T0/T1/T2, негайні), розподіл за типами порушень, географія по регіонах, рівні черги (Donut), якість моделі. **Без ПІБ.**
- **Черга реагування (`/queue`)** *[громада]* — список дітей за терміновістю; картка з **поясненням індексу терміновості** (які сигнали, з яких реєстрів, свіжість). Скоуп + ПІБ.
- **Профіль дитини (`/profile`)** *[громада]* — зібраний з реєстрів профіль: реєстри-силоси, хронологія сигналів, відвідуваність (графік зі зламом), **блок «Сімейні фактори ризику»** (§7), **блок «Слід в Естонії»** (§8).
- **Навантаження служб (`/caseload`)** + **Кабінет фахівця (`/my-queue`)** — розподіл по працівниках за територією, особиста черга + рішення (взяти в роботу / підтвердити).
- **Приватність і якість (`/privacy`)** — рівні доступу 1/2/3, PSI-сигнал Рівня-1, метрики моделі — «чому системі можна довіряти».

## 7. Ризики (ТВОЯ частина — інтегрувати, не лишати «потім»)
На гілці `feature/estonia-crossborder-ui` вже є **повний стек**, портуй/дороби:

**Два виміри ризику дитини:**
1. **Власні сигнали дитини** — порушення (W1 переміщення, P1 насильство вдома, W3 поза освітою, W6 сирітство, F3 нехтування тощо), кожне з: severity × evidence (перетин реєстрів) × acuity (свіжість).
2. **Батьківські/сімейні фактори** (SUBSTANCE_FEATURES §1–4,8): кримінал батьків, ДН з боку опікуна, психіка/залежність (вплив на догляд), позбавлення/втрата батьків, економічний шок, насильство над сиблінгом.

**Зважування за СУТТЮ, не за фактом:** `value = база × сила_доказу × згасання_за_давністю × стосунок_кривдника × роль_дитини`.

**Запобіжники (обов'язково — це показує зрілість):**
- **Бідність ≠ ризик** (рахуй лише припинення допомоги, не статус);
- **брак даних = unknown**, не штраф (медтаємниця);
- toxic-trio — пояснювальний контекст, не автоматичний бал;
- усе **прозоро пояснюється** в UI (внесок кожного фактора + чому).

**Архітектурний шов (як в MVP):** ризики — це `Contribution[]` з тегом `dimension: "child" | "parental"`, які входять у наявну агрегацію скорингу (`top + 0.2·Σ`). Тоді навіть **немовля з кривдником-батьком** отримує пріоритет **до прямої шкоди** (превентивний кейс — сильний демо-аргумент).

**UI:** блок «Сімейні фактори ризику» в профілі (сила доказу, стосунок, давність), чіп «ризик від родини» в черзі, розподіл факторів на дашборді (агреговано для регіонала).

> Крос-реєстрове виявлення — **лише на рівні ССД** (єдина роль з інтегрованим Рівнем-2). Інші ролі — вертикальні зрізи або агрегати.

## 8. Крос-кордон UA↔EE (варіант C — ІНТЕГРАЦІЯ, не окремий екран)
Користувач **відхилив окремий екран** (як у `main`). Інтегруй у наявні екрани:
- **Профіль** → блок «Слід в Естонії» (присутність у EE-реєстрах RAHV/EHIS/TERVIS/SKAIS, щілина, оцінка PPRL-звʼязку, УНЗР vs isikukood);
- **Черга** → чіп 🇪🇪 на крос-кордонних кейсах;
- **Дашборд** → міні-блок «Крос-кордон» (скільки видно EE / звʼязано / у щілині).

Принцип: **isikukood ≠ УНЗР**, спільного ключа немає → лише імовірнісний PPRL «сигнал збігу», без передачі ПІБ. (Деталі — `docs/ESTONIA_LEGAL.md`, `docs/ESTONIAN_REGISTRY_ACCESS_FROM_UA.md`.)

## 11. Чого уникати / на що зважати (уроки з розробки)
- **Реалізм даних.** Жодних логічних нісенітниць (напр. «запис про народження на 10 років пізніше народження», школа для дитини <6 р., порушення без доказів). Дзеркаль логіку бекенду (severity/scoring/tiers/caseload).
- **Не «ші-дизайн».** Стримано, по-державному.
- **Без жаргону.** Кожен термін — простими словами.
- **Усе пояснювати** (прозорість > «магія балу») — це і вимога приватності, і довіра журі.
- **Демо має працювати офлайн** (моки), без падінь у dev і prod.
- Next 16: middleware = **`proxy.ts`**; якщо ламається після перейменувань — `rm -rf frontend/.next` і перезапуск.
