# Ролі × Реєстри × Доступ — хто що бачить і чим користується

> Виведено з `docs/CHALLENGE_REQUIREMENTS.md` (аркуш «Role», 9 ролей) + наших реєстрів (`docs/REGISTRY_SCHEMAS.md`) + правового режиму (`docs/ACTORS_AND_REGISTRY_ACCESS.md`, `docs/ESTONIAN_REGISTRY_ACCESS_FROM_UA.md`).
> Принцип: **сегрегація даних за роллю** — лікар не бачить суд, школа не бачить кримінал, батьки — лише свою дитину, поліція — кримінал/ДН. Крос-реєстрове виявлення робить лише ССД-рівень.
>
> **Звірено адверсарним рев'ю** (workflow `access-matrix-review`, 12 агентів, проти обох правових документів + реального права UA/EE). Виправлено: 2 критичні пробої стін (ЄРДР, eHealth), 5 крос-вертикальних over-grants поліції/Мінʼюсту, інверсію рівнів, колізію назви CBI, фрейминг EE; додано вимір території, read/feed-розділення, правило «батько-кривдник». Деталі змін — `docs/ACCESS_MATRIX_REVIEW.md`.
>
> **Канонічна шкала рівнів (за ACTORS §4):** **1 = публічні/слабочутливі · 2 = персональні з обмеженим доступом · 3 = особливо чутливі / спецкатегорії / таємниця слідства / медтаємниця.** (Раніше матриця/код використовували обернену шкалу — скасовано; код `registries.py` realign-ується до цього канону.)

## 0. Три відношення до системи (хто такий «користувач»)
| Тип | Хто | Роль у системі |
|---|---|---|
| **Оператор** | Фахівець ССД (Ф-Г/Ф-М/Ф-О) | єдиний інтегрований крос-реєстровий доступ **у межах своєї території**; бачить профіль + ред-флаги; веде кейс |
| **Вертикаль (джерело+отримувач)** | Лікар (МОЗ), Школа (МОН), Поліція (МВС), Мінʼюст, ПФУ | бачить ЛИШЕ свій реєстр; живить дані; отримує адресний сигнал/повідомлення |
| **Кінцевий споживач** | Батьки (додаток ЦШД), Наглядач/Аналітик (НСССУ/Мінсоц), Публіка | своя дитина / агрегати / знеособлене / публічно-відкрите |

**Виноски:**

2. ✅ — нотаріуси Мінʼюсту отримують повні дані/чинність документів з ЄДДР через e-Notary (ПКМУ №700 від 11.07.2023). Раніше було 🔆 (недогрант).
3. 🔆 = верифікація УНЗР/чинності документа (ПКМУ №784/2017; ЗУ №2464-VI ст.20) — НЕ «повний за ухвалою».
4. 🔆※ = **de lege ferenda**, feature-flag OFF (сьогодні фактично ✋). Чинної підстави доступу ССД до ЕСОЗ немає (ACTORS §4.2); медтаємниця Основи 2801-XII ст.39-1/40. Зміст медзаписів — заборонено завжди. Жива траєкторія — лікар→ССД (обовʼязкове повідомлення).
5. 🌐 = законно публічно-відкрите: ЄДРСР — по-рішенню, знеособлено (ЗУ №3262-IV; усиновлення взагалі не вноситься); ДРРП — по-обʼєкту/особі, ідентифіковано+платно (ЗУ №1952-IV ст.32/34); ЄДЕБО — лише відкритий шар реєстру субʼєктів/документів, НЕ PII учня.
6. ✋ (було 🔆) — крос-вертикальний витік: поліція = кримінал/ДН; інформація про відсутність/статус надходить як вихідне повідомлення школи/ССД (емітер ПКМУ №684/658-2018), а не як читання/сигнал поліції; для ЄІАС-Діти/Діти-війни немає підстави.
7. ✋ (не 🏠) — правило «батько-кривдник»: власна-дитина пригнічена у кейс/кримінал/ДН/гаряча-лінія рядках (батько може бути фігурантом; заявник захищений конфіденційністю).
8. ✋ (було 🔆) — **КПК ст.222**: навіть факт/статус досудового розслідування — таємниця слідства; жодного автоматичного «є/нема». Лише через слідчого/прокурора з письмовим дозволом (процесуальний канал, push).
9. 📊 — лише якщо агрегат формує/видає держатель (Офіс ГП / МВС), не аналітик «Ластівки» з row-level; обовʼязкове придушення малих клітинок.
10. ✅ (було 🔆) — органи/заклади охорони здоровʼя та освіти — прямо названі субʼєкти реєстру ДН (ЗУ №2229-VIII ст.15-16; ПКМУ №234/2019), у межах інтересів постраждалого; розмежувати read vs write.
11. ✅ — підстава ПКМУ №121/2011 (через підрозділ місцевої держадміністрації); мінімізація, спецкатегорія ЗУ 2297-VI ст.7.
12. ✅ (було 🔆), скоуп до освітніх полів — МОН — названий користувач і постачальник даних ЦБІ для інклюзії; без медичного діагнозу.
13. ✅ — рольовий доступ до підсистем/полів у межах функції (ПКМУ №404/2021), НЕ повний доступ до всього хабу.
14. ✋ (було 🔆) — СКАЙД ідентичність/власник/акт [потребує звірки]; немає підтвердженої підстави авто-сигналу до ССД. Відновити 🔆 лише за наявності названого акта МВС.
15. ✅ (було 🔆) — ДРРП відкритий платний реєстр; ст.32 дає ОМС/опіці та Нацполіції безоплатний прямий доступ у звʼязку з повноваженнями; для поліції — purpose-bound + аудит.
16. 🔆 — re-based: ССД/соцзахист звільнені від судової ухвали (ЗУ №2464-VI carve-out для органів праці/соцзахисту); маркер зайнятості/доходу через ЄІССС, НЕ «повний за ухвалою».
17. ✅ — це **інтейк/feed** (вхідний канал), а не крос-реєстрове читання; скоуп території.
18. 🔆 = оператор PUSH-передає звернення з ознаками кримінал/ДН/загрози (ЗУ №2229-VIII; ПКМУ №800/2018) — НЕ запит, НЕ розблокування за ухвалою, БЕЗ широкого читання лог-журналу.
19. 🌐 = PPRL крос-кордонний сигнал «ймовірний збіг/немає» (тригер до Гаага-1996 через Justiitsministeerium → SKA/KOV); НЕ доступ до реєстру; повний контент — лише міждержавним каналом, НЕ за ухвалою UA. Збіг — лише на ідентифікаційних квазі-ідентифікаторах.
20. ✋ для рутинного PPRL; медичне (GDPR ст.9) — лише невідкладно (Гаага ст.31/32 + GDPR ст.49(1)(f), через лікаря/SKA). НЕ частина рутинного сигналу.
21. ✋, без PPRL — кримінальні дані (GDPR ст.10; Дир. 2016/680); лише правоохоронний/Interpol канал.

## 2. Що кожна роль РОБИТЬ (призначення)

### Фахівець ССД (Ф-Г громада / Ф-М місто / Ф-О область) — ОПЕРАТОР
- **Доступ:** інтегрований крос-реєстровий (Рівень-2), **у межах своєї території** (Ф-Г — громада, Ф-М — місто, Ф-О — область) — це **обовʼязковий query-time фільтр**, не nationwide.
- **Стіни:** **ЄРДР — ✋** (нічого автоматично; лише процесуальним каналом за письмовим дозволом слідчого/прокурора, КПК ст.222). **eHealth — 🔆※ feature-flag OFF** (підстави немає, de lege ferenda); жива траєкторія — лікарське повідомлення. **СКАЙД — ✋** (непідтверджений акт).
- **Що юзає:** зібраний профіль дитини, чергу ред-флагів/відхилень, контрольні візити, строки реагування (585), сімейні форми влаштування; ДРРП ✅ (захист майнових прав), ПФУ 🔆 (маркер доходу через ЄІССС).
- **Це наш головний користувач** — під нього зроблено чергу/тріаж/профіль/caseload.

### Лікар (вертикаль МОЗ)
- **Доступ:** eHealth/ЕСОЗ (повна медкарта, **медтаємниця**), ЦБІ_DISABILITY, **Реєстр ДН ✅** (у межах інтересів постраждалого — заклади охорони здоровʼя є названим субʼєктом, ЗУ 2229-VIII). **НЕ бачить** ЄРДР/суд-зміст/соц/освіту.
- **Що юзає:** медичні події, вакцинація, декларація. **Обовʼязок (feed-OUT):** повідомити ССД+поліцію про ознаки насильства (≤24 год / ≤3 год у закладі) → тоді сигнал іде в ССД.

### Школа (вертикаль МОН)
- **Доступ:** ЄДЕБО, АІКОМ, **ЦБІ_DISABILITY ✅** (лише освітні поля для інклюзії, без діагнозу), **Реєстр ДН ✅** (victim-scope, заклади освіти — названий субʼєкт). **НЕ бачить** медичний зміст/суд/кримінал.
- **Що юзає:** зарахування, відвідуваність, успішність. **Обовʼязок (feed-OUT, НЕ читання поліцією):** повідомити ССД+поліцію (ПКМУ 684 — 10-й робочий день відсутності; ПКМУ 658/2018 — ≤3 год). Це **емітер**, а не доступ поліції до АІКОМ/ЄДЕБО.

### Поліція / ювенальна превенція (вертикаль МВС)
- **Доступ:** ЄРДР, реєстр ДН, ІКС СКАЙД, ЄДДР, ЄДРСР; ДРРП ✅ (purpose-bound до кримінал/ДН-провадження + аудит). **НЕ читає** освіту (АІКОМ/ЄДЕБО), соц-вертикалі (Діти війни, ЄІАС Діти), медичне — лише отримує **вихідні повідомлення** від школи/лікаря/ССД.
- **Гаряча лінія:** лише **push-передача** звернень з ознаками кримінал/ДН (не широке читання лог-журналу).
- **Що юзає:** кримінальні провадження, домашнє насильство, профілактичний облік, заборонні приписи.

### Мінʼюст / ПФУ (інші вертикалі)
- **Мінʼюст:** ДРАЦС, ЄДРСР, ДРРП, **ЄДДР ✅** (нотаріуси через e-Notary, ПКМУ 700/2023). **НЕ** ЄРДР, **НЕ** ЄІАС Діти (кожен — лише свій зріз). **ПФУ:** реєстр застрахованих/виплати, ЄІССС (названий субʼєкт, рольово), ЄДДР-верифікація.

### Батьки (Б — додаток ЦШД)
- **Доступ:** **лише власна дитина** (через Дія.Підпис). Статуси, нагадування, виплати, поради.
- **Правило «батько-кривдник» (guard, не хардкод):** власна-дитина **пригнічена (✋, не 🏠)** на рядках ЄІАС-Діти/кейс, ЄРДР, Реєстр ДН, СКАЙД, гарячі лінії; і умовно — там, де батько є фігурантом ДН/кримінального провадження.

### Наглядач/Аналітик (Н-Р НСССУ / Н-Н Мінсоц)
- **Доступ:** агреговані KPI, якість даних, контроль користувачів, інтернатний блок; персональні дані — **лише за службовою потребою**.
- **Стіни в агрегатах:** ЄРДР/СКАЙД — 📊 лише якщо агрегат **формує/видає держатель** (Офіс ГП/МВС), не row-level reconstruction; обовʼязкове **придушення малих клітинок** (re-identification на ювенальних даних).

### Адміністратор / Публічний
- **Адм:** користувачі/ролі/інтеграції/аудит — **без доступу до кейс-даних**. **Публіка:** знеособлені агрегати (📊) + законно **публічно-відкрите (🌐)**: ЄДРСР (знеособлені рішення), ДРРП (ідентифікований витяг, платно), ЄДЕБО (відкритий шар субʼєктів/документів).

## 3. Як це лягає на наш код (поточний стан + дії з рев'ю)
- **Маємо:** `access` (int) на кожному реєстрі; privacy-шар (Рівень-1 → PSI-сигнал); одну модель кейсворкера + caseload-розподіл за територією.
- **Треба (з рев'ю, ДО збірки рольового кабінету):**
  1. **Канон рівнів:** перевернути `access` int до шкали ACTORS (1=публічне … 3=особливо чутливе): `EHEALTH 1→3`, `ERDR 1→3`, `EDRSR 3→1`, `TERVIS 1→3`; усі поточні `2` лишаються. **КРИТИЧНО:** спершу **аудит кожного гілкування на `access==1`** (PSI-гейт у `privacy`/`detection`; `REG_ACCESS == 1` у `app/api.py` timeline, `app/dashboard.py`) і перенацілити на `==3`, інакше privacy-шар мовчки інвертується. Безпечніше — гілкувати на новому `access_class` (рядок), не на int.
  2. **`access_class`** (рядок) + **`cluster`** (C0..C7/EE) на кожному реєстрі: WALLED={EHEALTH, ERDR, TERVIS, KaristEE}, OPEN={EDRSR, ДРРП, ЄДЕБО-public}, GATED-L2=решта.
  3. **Перейменування:** код `CBI` → `CBI_DISABILITY` (інвалідність, ПКМУ 121/2011, лікар ✅); окремий код `SJO_NOTIFY` для потоку повідомлень про дітей у СЖО (ПКМУ 800/2018, ССД-only/hotline-fed, НЕ лікар-readable). Оновити `REG_BY_CODE`/`EMITTERS`.
  4. **EE:** розгорнути єдиний EE-рядок на RAHV/EHIS/SKAIS (🌐 PPRL-сигнал) + TERVIS (✋ рутинно) + Karistusregister (✋, без PPRL); EE `access` int НІКОЛИ не інтерпретувати як UA-side readable grant — лише PPRL-тригер.
  5. **`roles.py`** (новий): (a) 9 рольових кодів як параметр, фільтр `rows_by_reg` за виправленою матрицею; (b) вертикальні вікна; (c) **територіальний фільтр** (oblast/місто/громада) на query-time; (d) батьківський режим Дія.Підпис + **guard «батько-кривдник»**; (e) **read vs feed** розділення (обовʼязкові повідомлення = емітери); (f) EE → лише PPRL-тригер, ніколи readable; (g) ЄРДР push-only / eHealth feature-flag OFF; (h) аналітик 📊 з held-released агрегатів + small-cell suppression; (i) 🌐 public-open рендер для ЄДРСР/ДРРП/ЄДЕБО.
- **Гарантії (підтвердити в коді):** жоден емітер/PSI-шлях не віддає ЄРДР або eHealth «є/нема» булеан до ССД/Мінʼюсту; гаряча лінія — push-only (`action_taken='передано до ССД та поліції'`), без широкого читання поліцією.

## 4. Ключовий принцип (для журі й для коду)
**Крос-реєстрове виявлення червоних прапорців — виключно на рівні ССД** (єдина роль з інтегрованим Рівнем-2, у межах території). Усі інші ролі — вертикальні зрізи, агрегати або публічно-відкрите. Найчутливіше (медичне, кримінал) перетинає межу лише як **сигнал через PSI** — а **ЄРДР і eHealth не перетинають навіть як сигнал** (стіни КПК ст.222 / медтаємниці; eHealth — de lege ferenda). Крос-кордон EE — лише **PPRL-сигнал збігу**, не доступ. Це і є юридично коректне розмежування «лікар ≠ суд», звірене адверсарним рев'ю.
