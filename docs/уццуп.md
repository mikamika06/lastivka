A. Інфраструктура (просте, тут нема магії)

DuckDB — це база даних, але «вбудована», як SQLite, тільки заточена під аналітику. Різниця:
- Postgres/MySQL — сервер: окремий процес, треба підняти, підключитись по мережі. Заточені під багато дрібних транзакцій (OLTP — «додай рядок», «онови запис»).
- DuckDB — немає сервера. Це бібліотека: pip install duckdb, і вона працює прямо всередині твого Python-процесу. Колонкова (OLAP) — блискавична на JOIN, GROUP BY, агрегаціях по мільйонах рядків. Читає CSV/Parquet напряму однією командою.

B. ЯДРО: як зматчити дитину між базами, не розкриваючи PII

Це найважливіше. Будую ланцюжком, бо кожен крок спирається на попередній.

1. Bloom filter (структура даних)

Уяви масив із, скажімо, 1000 бітів, усі нулі. Є k хеш-функцій. Щоб додати елемент — проганяєш його через усі k хешів, кожен вказує на позицію, ставиш там 1.
- Перевірка «чи є елемент»: хешуєш, дивишся біти. Якщо хоч один 0 — точно немає. Якщо всі 1 — напевно є (можливі хибнопозитивні, бо біти могли поставити інші елементи).
- Тобто Bloom filter — компактний імовірнісний «набір». Сам по собі це класика з мережних кешів.

2. q-grams (робимо порівняння нечітким)

Розбиваєш слово на підрядки довжини q. Для q=2 (біграми) ім'я olena → ol, le, en, na.
- Навіщо: схожі рядки ділять багато спільних q-gram. irina → {ir, ri, in, na}, iryna → {ir, ry, yn, na}. Спільні: ir, na. Не точний збіг — але й не нуль. Ось так ловимо опечатки й варіанти написання.

3. Bloom filter + q-grams = нечіткий приватний відбиток

Тепер головний трюк PPRL: замість додавати в Bloom filter ціле ім'я — додаєш його q-grams.
- olena → засіваєш у фільтр біти для ol, le, en, na.
- olenna (опечатка) → засіває майже ті самі біти.
- Порівнюєш два фільтри просто рахуючи спільні 1-біти. Багато спільних → та сама людина.

4. Dice similarity (метрика збігу фільтрів)

Формула: Dice = 2c / (a + b), де a = к-сть 1-бітів у фільтрі A, b = у фільтрі B, c = спільних 1-бітів. Результат 0..1.
- Та сама дитина з різним написанням → Dice ~0.85+. Різні діти → ~0.2. Ставиш поріг.

5. Чому це приватно (keying / HMAC-сіль)

Якщо просто хешувати, атакувальник міг би взяти словник імен, прохешувати й зіставити. Тому кожна установа засіває біти з секретним спільним ключем (HMAC). Без ключа фільтр — просто шум.
- На вихід установи віддають лише бітові масиви, не імена, не дати. Друга сторона бачить біти, рахує Dice — і ніколи не бачить чужий plaintext. Ось чому це «privacy-preserving».

6. CLK (Cryptographic Long-term Key)

Метод Шнелля: не роби окремий фільтр на ім'я, окремий на дату — запихай усі ідентифікатори (ім'я + прізвище + дата народження + …) в ОДИН фільтр на запис. Тоді одне порівняння Dice = схожість цілих записів. Це і є CLK.

7. Бібліотеки clkhash + anonlink

Не пиши це руками. CSIRO Data61 зробили:
- clkhash — на вхід записи (CSV), на вихід CLK-фільтри за схемою, яку ти задаєш (які поля, який q, скільки бітів).
- anonlink — бере два набори CLK, рахує всі Dice, і розв'язує задачу зіставлення (хто з ким у парі) оптимально. Тобто готовий PPRL-движок.

8. Атаки і захист (щоб витримати питання журі)

Christen та ін. показали: часті імена дають часті бітові патерни → частотний криптоаналіз може частково відновити дані. Контрзаходи, які треба знати:
- Balanced Bloom filter — додають дзеркальні біти, вирівнюють частоти.
- BLIP — підмішують випадковий шум до фільтра (по суті диференційна приватність): трохи псує точність, сильно псує атаку.
- Record-specific salting — індивідуальна сіль.
- Один слайд «ось вектор атаки, ось наш захист» = саме те, що люблять естонці (Guardtime-ДНК).

C. Шар під PPRL: класичний record linkage

PPRL — це приватна надбудова над звичайним record linkage (зіставленням записів про одну сутність). Концепти, які треба розуміти, бо вони керують і приватною версією:

- Fellegi-Sunter — імовірнісна модель зіставлення. Для кожної пари записів рахує дві ймовірності на кожне поле: m = «поля збіглись, якщо це справді одна людина», u = «поля збіглись випадково, якщо це різні люди». Комбінує в одну вагу → поріг → match / non-match / «можливо». Це стандарт галузі з 1969.
- Blocking — наївно треба порівняти N×M пар (мільйон на мільйон = трильйон, нереально). Blocking спершу грубо групує кандидатів (напр. «той самий рік народження» або «та сама перша літера») і порівнює лише всередині блоку. Прибирає 99% пар.
- LSH (locality-sensitive hashing) — розумніший blocking: хеш, що з високою ймовірністю кидає схожі записи в один кошик. Для Bloom-фільтрів є спеціальний LSH (MinHash) — ним anonlink і пришвидшується.
- Sorted neighborhood — ще один blocking: відсортуй за ключем, веди ковзне вікно.
- String similarity (бібліотека jellyfish): Jaro-Winkler (для імен — нагороджує спільний префікс), Levenshtein (мінімум правок, щоб з одного слова зробити інше), q-gram/Dice. Знати, коли який.
- Splink (Мін'юст Британії) — сучасна бібліотека, що робить Fellegi-Sunter із EM-навчанням (тобто оцінює m/u без розмічених даних, unsupervised) і працює прямо на DuckDB. Це я й раджу для непривітного шару / для бейзлайна.
- recordlinkage — простіша Python-бібліотека, добра для швидкого прототипу, поки не дійшли до Splink.

---
D. Синтетичні дані + транслітерація

Без даних від організаторів — ти їх генеруєш сам, і це навіть краще контрольовано:
- Faker — бібліотека, що генерує фейкових людей (імена, дати) у локалях uk_UA, et_EE, pl_PL.
- Ground truth — ти знаєш, який запис у базі А = який у базі Б (одна дитина), і навмисно «губиш» частину (зник з усіх сервісів). Це дає тобі точні precision/recall і священний слайд валідації.
- Corruption — навмисно псуєш копії: опечатки, інші формати дат, пропуски. Імітує реальний бруд.
- Транслітерація — головна крос-кордонна проблема: Олександр латиницею пишуть Oleksandr (УА), Alexander (виш), Oleksander (помилка). Одна дитина — три написання в трьох базах. Точний матч (PSI) тут здохне, а q-gram/Bloom — ні. Бібліотеки: transliterate, unidecode.
- GeCo / Febrl — дослідницькі генератори з вбудованим corruption, але старі (Python 2). Раджу свій corruptor на 50 рядків — контролюєш сценарії.

---
E. Другий движок: виявлення «дитина випала»

Логіка: дитина була в сигналах (ходила в школу/до лікаря) → зникла з усіх → у новій країні не з'явилась.
- Change-point detection — виявлення моменту, коли статистичний режим часового ряду змінився. Бібліотека ruptures (Truong).
- CUSUM — «cumulative sum», класична карта контролю: накопичуєш відхилення, спрацьовує, коли сума перевищить поріг.
- Bayesian online changepoint (Adams–MacKay 2007) — імовірнісний онлайн-варіант.
- Простіший і чесніший підхід: модель очікуваної каденції подій (renewal process) — «дитина зазвичай має шкільний сигнал що 3 місяці» → флаг, коли пропуск > поріг, з поправкою на крос-кордонний перехід. Часто цього досить, не ускладнюй.

---
F. PSI — і чому ми його НЕ беремо

PSI (Private Set Intersection) — криптопротокол, де дві сторони знаходять спільні елементи двох множин, не розкриваючи решту. Реалізації: Diffie-Hellman-based, OPRF (напр. KKRT).
- Проблема: PSI дає точний перетин — Oleksandr ≠ Alexander, для нього це різні люди. На транслітерації він сліпне.
- Тому наш свідомий вибір — Bloom-filter PPRL: він і приватний, і нечіткий. Це сильна, готова відповідь на питання журі «а чому не PSI?».

Повне меню технологій по шарах

Для кожного шару даю lean → serious → maximal, щоб ти сам обрав глибину. Нічого не викидаю.

1. Сховище / реєстри

- DuckDB — вбудована OLAP, блискавична аналітика, нуль setup. Lean.
- PostgreSQL — справжній сервер-реляційка. Дає окремі бази на установу (точно як реальні держреєстри), ролі/доступи, аудит-логи. Serious. Плюс екстеншени:
  - pg_trgm — нечіткий пошук по триграмах прямо в БД,
  - pgcrypto — хешування/шифрування в БД,
  - PostGIS — гео (пункти допомоги в застосунку).
- Дві окремі інстанції Postgres, що говорять між собою — імітують X-Road/Trembita-федерацію, де кожна установа тримає свою БД. Maximal і прямо в естонську ДНК.

2. Обробка даних / де крутяться моделі

- pandas — стандарт.
- Polars — Rust-движок, у рази швидший, сучасний. «Proper» вибір.
- PySpark — справжній скейл (мільйони записів). Splink і Zingg працюють на Spark → твоя відповідь журі «чи масштабується на національні реєстри». Запускати для демо не обовʼязково, але мати в архітектурі — сильно.

3. Matching / entity resolution (модель №1)

- Splink (Мін'юст Британії) — production-grade імовірнісне зіставлення (Fellegi-Sunter + EM), на DuckDB/Spark. Національний масштаб у реалі. Головний серйозний інструмент.
- Zingg — ML entity resolution на Spark, production.
- dedupe — active-learning дедуплікація (Python).
- Власний класифікатор: XGBoost/LightGBM/CatBoost на векторі порівнянь полів → «та сама / різні». Це і є «натренувати модель».
- Нейронний матчинг (модерн, крос-мовний): мультиязичні ембединги імен — LaBSE, multilingual-E5 через sentence-transformers. Реально допомагають з Oleksandr↔Alexander, бо ловлять семантику, не лише літери. Сильний сучасний кут.

4. Privacy / crypto шар (наш диференціатор — тут не економимо)

- clkhash + anonlink (CSIRO) — Bloom-filter PPRL. Прагматичне ядро.
- Справжня PSI: OpenMined PSI, Google Private Join & Compute, Microsoft APSI — якщо хочеш точну криптографічну частину поряд із нечіткою.
- MPC (secure multiparty computation): Sharemind — це продукт Cybernetica, естонської компанії; згадка/демо = величезний локальний кредит довіри. Або MP-SPDZ (open-source).
- Гомоморфне шифрування: TenSEAL/Microsoft SEAL/OpenFHE — рахувати на зашифрованих даних. Важко, але навіть іграшкове демо вражає цю аудиторію максимально.
- Differential privacy: OpenDP / Google DP / PyDP — для хардення (BLIP) і приватної агрегованої статистики.

▎ Естонці винагороджують саме це. Sharemind + PSI/PPRL — пряме влучання у Guardtime-прецедент.

5. Risk / dropout (модель №2)

- LightGBM/XGBoost/CatBoost — робоча конячка табличного скорингу ризику.
- lifelines — survival analysis (час-до-випадіння). «Серйозна» постановка.
- ruptures — change-point у потоці сигналів.
- Граф: Neo4j (graph-БД, вражає — «моделюємо безпекову мережу дитини графом»), або NetworkX (просто), або PyTorch Geometric / DGL для GNN.
- Isolation Forest / аномалії — для незвичних патернів.

6. API / бекенд

- FastAPI + Pydantic — сучасний async Python-API, стандарт.
- gRPC — якщо хочеш справжній service-to-service / федеративний вайб між установами.

7. UI / дашборд + застосунок

- Streamlit / Gradio — дашборд кейсворкера чистим Python (без фронтендера).
- Plotly Dash — кастомніше.
- React/Next.js — якщо зʼявиться фронтендер.
- Застосунок-мокап: Figma; якщо живий — Flutter/React Native.

8. MLOps (якщо хочемо «зробили як належить»)

- MLflow — трекінг експериментів + реєстр моделей. Сигнал «ми інженери, не демо-стяжка».
- Weights & Biases — те саме.
- DVC — версіонування даних/моделей.

9. Доставка / деплой

- Docker + docker-compose — підняти multi-service (дві реєстр-БД + crypto-шар + API + дашборд) однією командою. Тут це реально доречно — наочно показує федеративну архітектуру.
- Живий лінк для журі: HuggingFace Spaces (Streamlit), Render/Railway/Fly.io — щоб клікнули й воно працює. Реальний плюс.

❯ wandb це що таке та як це можна використати

Weights & Biases (wandb) — це хмарний сервіс (+ Python-бібліотека) для трекінгу ML-експериментів. Простими словами: коли ти ганяєш десятки прогонів моделі з різними параметрами, wandb автоматично записує що ти запустив, із якими налаштуваннями і що вийшло — і малює це в красивому веб-дашборді. Замість блокнота з метриками і скриншотів — одне посилання, де все.

Що саме він робить

┌─────────────────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│     Можливість      │                                                   Що дає                                                    │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Experiment tracking │ логуєш метрики (precision/recall, AUC, loss) — він будує графіки в реальному часі, поки модель тренується   │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Config logging      │ зберігає гіперпараметри кожного прогону → потім бачиш «прогін №14 з порогом Dice 0.82 дав найкращий recall» │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Порівняння прогонів │ таблиця/графіки всіх запусків поряд — видно, який кращий і чому                                             │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Artifacts           │ версіонує датасети й моделі (який датасет → яка модель → який результат)                                    │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Sweeps              │ автоматичний підбір гіперпараметрів (grid/random/баєсівський) — він сам ганяє прогони                       │
├─────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│ Reports             │ збираєш графіки в шарований веб-звіт — буквально готовий слайд для журі                                     │
└─────────────────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

Як виглядає в коді (мінімум)

import wandb

wandb.init(project="lastivka-matching", config={"q": 2, "dice_threshold": 0.82})

# ... тренуєш / ганяєш матчинг ...

wandb.log({"precision": 0.94, "recall": 0.89, "auc": 0.97})
wandb.finish()

Усе. Далі заходиш на wandb.ai → бачиш графіки, порівняння, історію. Працює на безкоштовному тарифі для особистих/невеликих проєктів.

Як це використати конкретно в Ластівці

1. Калібрування матчингу — ти крутиш q-gram, розмір Bloom-фільтра, поріг Dice. Кожен прогін → wandb. Наприкінці одразу бачиш, яка комбінація дає найкращий precision/recall на твоєму synthetic ground truth.
2. Risk-модель (LightGBM) — логуєш фічі, гіперпараметри, AUC/calibration по прогонах. Sweeps сам підбере найкращі параметри, поки ти спиш.
3. Священний слайд валідації — твій графік «підсадили 50 втрачених → знайшли 47» можна вивести прямо з wandb Report як готову, шаровану візуалізацію для пітча. Журі клікає лінк і бачить, що ти реально валідував.
4. Командна синхронізація — ти і колега бачите ті самі прогони в одному дашборді, без «а яка версія в тебе локально».

Чесна оцінка для вашого випадку

- Плюс: малий поріг входу (~10 рядків коду), і дає сильний сигнал журі «ми працювали як інженери, а не зліпили демо». Звіт-лінк = безкоштовний професійний слайд.
- Мінус: для 72-год хакатона це nice-to-have, не критичний шлях. Якщо часу обмаль — твій ground-truth-графік можна намалювати й у matplotlib за 5 хвилин.
- wandb vs MLflow: wandb — хмарний, гарніший дашборд із коробки, кращий для шаринга й командної видимості. MLflow — self-hosted/локальний, безкоштовний без обмежень, але менш «гарний». Для хакатона з акцентом на шаровану візуалізацію журі — wandb зручніший.

Порада: під час підготовки ці 4 дні — постав wandb одразу, коли почнеш калібрувати матчинг. Звикнеш, і на хакатоні воно вже працює; не встигнеш докрутити до пітча — нічого не втрачаєш.

❯ окей в проді як воно має працювати

«В проді» MLflow перестає бути папкою на ноуті й розпадається на три окремі компоненти + реєстр моделей. Локально всі троє злиті в одну теку ./mlruns — у продакшені кожен стає окремим сервісом.

Життєвий цикл у проді (головне)

1. Train + log — тренувальна джоба (часто запущена через CI, не вручну) логує прогін на сервер: метадані → Postgres, артефакти → S3.
2. Register — найкращу модель реєструєш у Model Registry під імʼям, напр. lastivka-matching, з версією v7.
3. Promote — переводиш версію в Production (раніше через «stages» Staging→Production; у сучасному MLflow 2.9+ — через аліаси типу @champion/@challenger і теги). Це джерело правди «яка модель зараз бойова».
4. Serve — продакшен-сервіс не хардкодить шлях до файлу, а питає реєстр: «дай модель lastivka-matching@champion» і вантажить її. Або CI збирає Docker-образ серверу моделі й деплоїть у k8s.
5. Monitor → retrain — метрики в проді деградують → нова тренувальна джоба → новий прогін → нова версія → promote. Цикл.

Ключова ідея: Model Registry — це «прошарок між експериментом і продакшеном». Дата-саєнтист накидав 200 прогонів, але сервіс бере тільки ту версію, яку явно позначили champion. Ніхто не деплоїть «модель з ноута Васі».

- Не публічний AWS S3, а self-hosted MinIO + Postgres усередині держ-хмари (по суті всередину контуру Trembita/гов-інфраструктури) — дані не покидають державний периметр.

- Промоушн у Production проходить через людину/комітет (govt не пускає авто-деплой моделі, що стосується дітей) — MLflow це й підтримує (ручний transition + лог, хто й коли затвердив).

2. eHealth має ПОВНІСТЮ публічну схему пацієнта (github.com/ehealth-ua/api-spec) — і вона вже містить дитячі механізми, ідеальні для Ластівки: confidant_person (законний представник), BIRTH_CERTIFICATE, no_tax_id для дітей без ІПН, THIRD_PERSON авторизація для <14. Це наш якір форми — решту реєстрів моделюємо за його зразком.

3. Ключі матчингу й чексуми — підтверджені:
- РНОКПП: перші 5 цифр = дата народження, 9-та = стать, 10-та = контрольна (ваги [-1,5,7,9,4,6,10,5,7] mod 11 mod 10 — перевірив на двох реальних tax_id зі звіту, сходиться).
- УНЗР: 13 символів, перші 8 = дата; найкращий внутрішньо-український ключ (лінкує ДРАЦС↔ЄДДР↔eHealth), але кордон не переходить.
- isikukood: GYYMMDDSSSC + алгоритм контрольної цифри.

4. Транслітерація КМУ №55 — повна офіційна таблиця + позиційні правила (Є→Ye/ie, Ю→Yu/iu…). Це доказова база нашої проблеми: Олександр→Oleksandr, але в естонській базі Alexander/Aleksander → точний join падає → потрібен PPRL.

Головна ідея: реєстри як «логери» прихованого життя

Ми не малюємо рядки під готову мітку. Натомість:

1. Кожна дитина — агент із прихованим станом (де живе, чи в школі, чи активна медкарта, який стан сімʼї/безпеки).
2. Час тікає помісячно (24 місяці). Стан еволюціонує — у більшості дітей нічого не стається (фон).
3. Порушення = «шок», що міняє прихований стан (переїхав → displaced, почалось насильство → abuse_active, кинув школу → dropped).
4. На кожному кроці часу кожен реєстр сам вирішує, що залогувати, дивлячись лише на стан дитини: дитина в школі → ЄДЕБО/ІСУО пишуть відвідуваність; переїхала → ВПО пише запис, стара школа пише exit, eHealth закриває декларацію; кинула школу → тиша в освіті.

Тобто ми задаємо причину (шок), а симптом (крос-реєстровий патерн) виникає сам. Як Synthea для медзаписів, тільки крос-реєстрово.

Ризик — це прихований перехід стану, якого детектор не бачить. Він бачить тільки логи в силосах і має здогадатись:

ГЕНЕРАЦІЯ:  шок стану ──► розкидані події по реєстрах
ДЕТЕКЦІЯ:   події по реєстрах ──► інференс ризику   ← це наш продукт

Жодного шахрайства: god-view (справжня істина) детектору недоступний, реєстри розділені, матчинг і детекція — окремі етапи downstream.

Як ми це виявляємо і ранжуємо — алгоритм

Це не одна модель, а гібридний конвеєр (бо в дитячому захисті прозорість важливіша за +2% точності):

Виявлення (per child):
- A. Матчинг — зібрати дитину з силосів (УНЗР; fuzzy де нема).
- B. Правила — порахувати формули порушень (булеві перетини реєстрів) → які порушення є.
- C. Темпоральний шар — change-point / dropout (ruptures) на каденції сигналів → гострота/швидкість погіршення.

Тріаж (кого показувати першим) — багатофакторний urgency-score:

Терміновість(дитина) = max( IMMEDIATE_OVERRIDE,
        Σ_порушень [ тяжкість_типу × сила_доказів × гострота ] )  ×  вразливість
- Тяжкість типу — харм-вага: сексуальне/торгівля/депортація ≈ 1.0; нехтування/булінг ≈ 0.4.
- Сила доказів — глибина перетину: 1 реєстр < 3 реєстри.
- Гострота — свіжість + швидкість (різкість change-point, скільки місяців тиші).
- Вразливість (множник) — вік <6, інвалідність, уже в «Дітях», без опікуна, ВПО, + батьківські фактори ризику з нового скріна.
- IMMEDIATE_OVERRIDE — [1]-сигнали (сексуальне, торгівля, депортація, гостра травма+провадження) йдуть у топ без очікування перетину.

Вихід: три рівні черги — T0 «сьогодні» / T1 «цей тиждень» / T2 «спостереження», всередині рівня — ранжування за score, і кожен пункт із поясненням, які сигнали спрацювали (бо це рішення про дитину — чорна скриня неприпустима).

1. Ми не передбачаємо «чи battьки злочинці» — ми детектимо об'єктивну депривацію прав (дитина записана в школі чи ні; декларація активна чи ні). Це факт, не прогноз про особистість → уникаємо головної вади AFST.

2. Сила доказів (evidence) — множник за глибиною перетину

1 реєстр → 0.6 (може бути шум) · 2 → 1.0 · 3 → 1.3 · 4+ → 1.5. Бо, як ти й казав, доводить перетин, не одне джерело.

3. Гострота (acuity) — множник за свіжістю+швидкістю

різкий новий change-point (1–2 міс) → 1.5 · активне стабільне → 1.0 · хронічне без руху → 0.7 · покращується → 0.5. Сюди ж ruptures дає різкість зламу.

4. Вразливість (vulnerability) — множник, добуток факторів, cap 2.0

вік<6 ×1.3 · 6–10 ×1.15 · інвалідність ×1.3 · уже в «Дітях» ×1.2 · без опікуна ×1.4 · ВПО/ТОТ ×1.2 · батьківський фактор ризику (залежність/психіатрія/кримінал — через PSI-булеан) ×1.25.

Збірка

contribution(v) = severity[v] × evidence[v] × acuity[v]
raw = max_v(contribution) + 0.2 × Σ_решта(contribution)     # найгірше домінує, але кілька проблем додають
score = raw × vulnerability        (+ IMMEDIATE_OVERRIDE → T0)
Рівні черги: T0 «сьогодні» ≥1.8 або override · T1 «цей тиждень» 0.9–1.8 · T2 «спостереження» 0.35–0.9 · <0.35 — не флагаємо.

Чому це не «висмоктано»

Seed-ваги ставимо за харм-логікою, а пороги калібруємо на ground truth: крутимо так, щоб precision@T0 був високий (мало хибних тривог) при прийнятному recall. Це й є сенс мати синтетику з мітками — все логуємо в MLflow. Ваги в YAML-конфізі, експерт може правити без коду.

    """Офіційна транслітерація одного слова за КМУ №55."""
    w = word.lower()

    w = w.replace("зг", "z__g")  # тимчасовий маркер
    out = []
    for i, ch in enumerate(w):
        if ch == "_":
            continue
        if ch in ("ь", "'", "’"):  # мʼякий знак і апостроф опускаються
            continue
        if ch in _POSITIONAL:

    return res.replace("zg", "zgh") if "z__g" in word else _capitalize(res)
    return translit_official(word)
