# Як Ластівка виявляє ризики (по кожній базі + перетин)

## Головний принцип
**Жоден окремий реєстр не доводить порушення — доводить ПЕРЕТИН сигналів.** З кожної бази система
дістає не «сирий запис», а **похідний сигнал** (булеан або число). Порушення = коли сигнали з
**кількох баз збігаються в патерн**. Чим більше реєстрів підтверджують — тим вища впевненість.

Це 3 кроки: (1) кожна база → сигнали; (2) у часових рядах — точка зламу; (3) перетин сигналів → порушення.

---

## Крок 1. Що кожна база дає (сигнал, не дані)

| База | Сигнал, який дістаємо | Поле-джерело |
|---|---|---|
| **ДРАЦС** | смерть батька/матері | `act_type == "смерть"` |
| **ЄДЕБО** (освіта) | вихід зі школи | `study_status ∈ {transferred, expelled}` |
| **АІКОМ** (відвідуваність) | **сплеск пропусків**, падіння оцінок, комісія з булінгу, тиша=відсів | `missed_lessons_count` (ряд), `score_12` (ряд), `anti_bullying_commission`, останній `attendance_period` |
| **ЕСОЗ 🔒** (медицина) | хронік, пропущена вакцинація, травма (повторна), психічний стан, обрив декларації | `condition_code=chronic`, `immunization status=not_done`, `condition_category∈{trauma,psych}`, `is_repeat`, `declaration status=terminated` |
| **ВПО** | переміщення, чи поновлено школу/лікаря | `displacement_date`, `education_place`, `medical_needs` |
| **Діти війни** | депортація / переміщення / втрата батьків / **мілітаризація** / **поранення** | `status_category` (парситься) |
| **ЄІАС Діти (ССД)** | на обліку, **складні життєві обставини**, без супроводу | `child_status`, `difficult_life_circumstances=так` |
| **ЄРДР 🔒** (слідство) | стаття провадження: 149 торгівля / 152 сексуальне / 126-1 ДН | `preliminary_legal_qualification` (парситься) |
| **Реєстр ДН** | виклики поліції, форма насильства, **дитина-свідок** | к-ть записів, `form_of_violence`, `child_role=свідок` |
| **ЦБІ** | інвалідність | наявність запису |
| **ЄДРСР** (суди) | позбавлення батьківських прав | `case_category` містить «батьк» |
| **ДРРП** (житло) | відчуження житла дитини | `child_residence_alienation=так` |
| **ПФУ** | безробіття батьків | `employment_status_indicator=безробітний` |
| **Гарячі лінії** | звернення, тип насильства | наявність, `violence_type` |

> 🔒 ЕСОЗ/ЄРДР — **walled**: зміст (діагноз, фабула справи) НЕ читається; дістається лише факт-сигнал.

## Крок 2. У часових рядах — «точка зламу» (change-point)

Відвідуваність і оцінки — це **ряд за 24 місяці**. Система шукає, де середнє різко змінилось:
для кожного розрізу `k` рахує середнє ДО і ПІСЛЯ, бере найбільший стрибок.
- **Сплеск пропусків** = стрибок ≥ 4 уроки вгору (`absence_spike`).
- **Падіння оцінок** = стрибок ≥ 2 бали вниз (`gpa_drop`).
- **Гострота (acuity)**: злам в останні 4 міс = «гостре» (×1.5), давно = «хронічне» (×0.7).

Тобто система бачить не «дитина пропускає», а «дитина **раптом почала** пропускати» — це і є сигнал погіршення.

---

## Крок 3. Перетин сигналів → порушення (приклади)

Кожне правило — це **кон'юнкція сигналів з РІЗНИХ баз**. Приклади (спрощено):

**W3 «Поза освітою»** = шкільний вік І (ЄДЕБО: вихід зі школи АБО АІКОМ: тиша задовго до кінця).
→ освітня вісь, 1–2 бази.

**F3 «Нехтування»** = ЕСОЗ: пропущена вакцинація **І** АІКОМ: пропуски **І** ССД: складні обставини.
→ **три бази мусять збігтись** (медицина + школа + соцслужба). Бідність сама — не доказ.

**P1 «Фізичне насильство вдома»** = ЕСОЗ: повторні травми **І** (ДН: виклики поліції АБО ЄРДР: ст.126-1).
→ медичний сигнал **+ корроборація правоохоронцями**. Сама травма без поліції — не P1 (може бути випадкова).

**W8 «Обмеження медицини»** = ЕСОЗ: хронік + обрив декларації **І** ВПО: переміщення **І НЕ** ССД-обставини.
→ розрив медичного супроводу через переїзд, а не через бідність (інакше це соц-питання, не доступ).

**F5 «Свідок ДН»** = ДН: запис із `child_role=свідок` **І НЕ** (P1/F1/F6) — дитина бачить, але сама не жертва.

**W10 «Мілітаризація» / W4 «Поранення»** = Діти війни: відповідний статус → негайний рівень (T0).

**Запобіжник** (з аудиту): хвороба пояснює пропуски/невакцинацію — тому F3/F4 **не глушаться** на хворих
дітях (це дало б пропущений реальний ризик), а конфаундери на кшталт «зміна лікаря → W8» прибрано.

---

## Крок 4. Скільки це «важить» (корроборація)

Зібрані порушення проходять **пояснюваний скоринг**:
- **тяжкість** (severity) × **сила доказів** (1 база ×0.6, 2 ×1.0, 3 ×1.3, 4+ ×1.5) × **гострота**.
- **вразливість** (вік<6, інвалідність, ВПО, на обліку ССД) — множник.
- **рівень**: T0 (негайно) / T1 (тиждень) / T2 (спостереження). Негайні (торгівля, депортація,
  сексуальне, поранення, мілітаризація) — одразу T0.
- кожне рішення показує **які саме сигнали спрацювали** — не чорна скриня.

---

## Крок 5. Як це працювало б на РЕАЛЬНИХ даних

Логіка детекції **не змінюється** — змінюється, ЗВІДКИ беруться сигнали:

| Зараз (демо) | На реальних даних |
|---|---|
| 21 локальна SQLite-база | Кожен реєстр у власника (Мінʼюст, НСЗУ, МОН…), доступ через **Трембіту** (держшина) |
| matching читає рядки локально | Кожен реєстр віддає **той самий похідний сигнал через API** (напр. ЕСОЗ: `has_chronic: yes/no`, не діагноз) |
| Зв'язування по plaintext-УНЗР | **PSI/PPRL**: реєстри зіставляються за зашифрованими відбитками, без обміну ПІБ |
| Walled = редакція в коді | Walled = реєстр **фізично не віддає зміст** зі свого API, лише факт-булеан |
| Детекція в одному процесі | Детекція на **вузлі-агрегаторі** (MPC/TEE), що бачить лише сигнали, не сирі бази |

**Суть:** реальний реєстр віддає не «медкартку», а відповідь на конкретне запитання
(«чи є хронічне? чи пропущена вакцинація?») — рівно той сигнал, який потрібен правилу. Детекція
рахує перетин цих відповідей. Так зберігається і медтаємниця, і ефективність.

> `lastivka/scoring.py`; стіни — `lastivka/roles.py`; крипта — `lastivka/privacy.py`.
