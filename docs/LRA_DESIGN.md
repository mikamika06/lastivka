# LRA — Local Risk Agent: як працює під капотом по кластерах реєстрів

> Грунтується на `docs/ACTORS_AND_REGISTRY_ACCESS.md` (правовий режим, рівні чутливості, «Трембіта» система↔система, мінімізація, сигнал-замість-сирих-даних) та `docs/ROLE_ACCESS_MATRIX.md` (сегрегація за роллю). Лягає прямо на код: `detection.py:_signals` — це **вже централізована версія LRA**, розкладена тут на децентралізовані вузли.

---

## 0. Що таке LRA (одним абзацом + контракт)

**LRA (Local Risk Agent)** — легкий детектор, що стоїть **поруч із кожним реєстром-джерелом** (на боці володільця даних, за логічною межею «Трембіти»). Він **читає сирий запис лише локально**, зводить його до канонічних подій із 5 осями суті (severity / role / recency / recurrence / outcome), застосовує локальні правила свого кластера і **віддає назовні не дані, а сигнал**. Це не «новий канал доступу людини до реєстру», а реалізація принципу мінімізації (ст. 6 ЗУ № 2297-VI) і моделі «система ↔ система» (`ACTORS §3.3`, `§6`).

**Контракт LRA (інваріанти, які не порушуються в жодному кластері):**
1. **Сирі дані не залишають реєстр.** Назовні — лише `signal envelope` (див. §2). Діагноз, анамнез, зміст матеріалів слідства, текст рішення — ніколи.
2. **Псевдонім, не ідентифікатор.** Зв'язування йде за `pseudonym = HMAC_k(УНЗР)` / Bloom-CLK(імʼя+ДН), не за УНЗР у відкритому вигляді (`crossborder._clk`, `privacy.clk`).
3. **Доступ за класом (§3), не за бажанням.** LRA фізично не вміє віддати те, на що немає правової підстави: для `WALLED`-кластерів немає коду «pull → emit».
4. **Крос-реєстровий перетин — лише на рівні ССД** (`ROLE_ACCESS_MATRIX §4`). LRA вертикалі (лікар/школа/поліція) сигнал **лише віддає**; зводить його докупи один агрегатор.
5. **Журнал кожної емісії** (хто-вузол, коли, мета, які поля торкнуто, кому пішло) — `ACTORS §3.6`.

## 2. Signal envelope (єдиний формат, який віддає будь-який LRA)

```jsonc
{
  "pseudonym":      "b9f1…",      // HMAC_k(УНЗР) або Bloom-CLK; НЕ УНЗР
  "indicator":      "absence_spike",  // ім'я сигналу з detection.py:_signals
  "cluster":        "C3_education",
  "registry":       "AIKOM",
  "access_class":   "GATED-L2",   // OPEN | GATED-L2 | WALLED  (див. §3)
  "value":          true,          // булеан АБО скор; ніколи не сирий запис
  "severity":       0.0-1.0,        // вісь суті: тяжкість
  "role":           "victim|subject|context",  // вісь суті: роль дитини
  "recency_months": 3,             // вісь суті: давність
  "recurrence":     2,             // вісь суті: повторюваність
  "outcome":        "open|resolved|escalated",  // вісь суті: результат
  "onset_month":    14,            // для acuity (acute/active/chronic)
  "emitted_at":     "…",
  "audit_id":       "…"            // запис у журнал доступу
}
```

> 5 осей суті — це `docs/SUBSTANCE_FEATURES.md`: не «є кримінальна справа», а **що саме сталося, з ким, наскільки давно, скільки разів, чим скінчилось**. Сьогодні в коді частина осей згорнута (severity≈acuity, recurrence для травм/викликів); envelope — місце, куди вони виносяться явно.

## 3. Три класи доступу (як LRA гейтить себе) — виведено зі звіту

Звіт інвертує нумерацію рівнів (§4 «1=публічне», матриця «1=ухвала»). Щоб не плутати цифри, LRA оперує **класами поведінки**, а не номерами:

| Клас | Що означає для LRA | Реєстри (наш код) | Правова прив'язка |
|---|---|---|---|
| **OPEN** | LRA читає вільно, може віддавати і сигнал, і контекст | ЄДРСР, ДРРП (частково) | ЗУ № 3262-IV (відкритий, знеособлення); ЗУ № 1952-IV ст. 32 |
| **GATED-L2** | LRA рахує локально й віддає **сигнал** через «Трембіту» з журналом і роллю; повний зміст — окремим запитом органу | ДРАЦС, ЄДДР, ЄДЕБО, ІСУО/АІКОМ, ВПО, Діти війни, ЄІАС «Діти», ЄІССС, ЦБІ, ПФУ, ДН, СКАЙД, гарячі лінії | ст. 6, 11 ЗУ № 2297-VI; ПКМУ № 606/2016 (Трембіта) |
| **WALLED** | LRA **не має** механізму «pull → emit». Сигнал можливий лише **push-ем** від уповноваженого суб'єкта по його процедурі | ЕСОЗ (немає правової підстави — *de lege ferenda*), ЄРДР (ст. 222 КПК — навіть факт = таємниця слідства) | Основи № 2801-XII ст. 39-1/40; ст. 222 КПК; `ACTORS §3.5, §4.2` |

**Наслідок, який треба показати журі:** порушення, що залежать від WALLED-реєстрів (W7 торгівля, F6 секснасильство, ЄРДР-нога P1, ЕСОЗ-нога W8/W2), **не виявляються автоматичним перетином LRA**. Вони заходять у систему лише легальним push-каналом: лікарське обов'язкове повідомлення (≤24 год) або дозвіл слідчого/прокурора. LRA для цих кластерів — приймач авторизованого push, а не сканер.

## 4. Кластери реєстрів і робота LRA в кожному

Скорочення колонок: **Read** = сирі поля, які LRA бачить локально · **Local-only** = порушення, які кластер вирішує сам (перетин не потрібен) · **Contributes** = сигнали в крос-реєстровий перетин (§5).

### C0 — Хребет ідентичності та сім'ї (ЄДДР, ДРАЦС) · GATED-L2
| | |
|---|---|
| **Роль LRA** | Не рахує ризик — дає **псевдонім для зв'язування** і **ребра сімейного графа** (народження, смерть, шлюб/розлучення, батьківство). Це «join-шар» для всіх інших кластерів. |
| **Read** | УНЗР, ПІБ, ДН, стать, акти цив. стану (`act_type`), звʼязки батьки↔дитина↔сиблінги |
| **LRA робить** | `pseudonym = HMAC_k(УНЗР)`; emit ребер графа як псевдонім↔псевдонім; `parent_death = act_type=="смерть"` |
| **Local-only** | — (контекст, не порушення) |
| **Contributes** | `parent_death` → у W6 (сирітство). Псевдонім → у **всі** перетини |
| **Гейт** | ССД має доступ; зміст актів — за запитом органу. Дитина в усиновленні: рішення **не вносяться** до ЄДРСР (`ACTORS §4.2`) — граф не повинен їх відновлювати |

### C1 — Ядро захисту дитини / домашній вузол ССД (ЄІАС «Діти»/DITY, «Діти війни»/CHILDWAR, ЦБІ) · GATED-L2, прямий доступ ССД
| | |
|---|---|
| **Роль LRA** | **Домашній вузол.** Тут ССД — законний володілець (`ACTORS §2.5`), тому LRA найбагатший і **тут же стоїть агрегатор** (§5). Стан кейсу живе тут. |
| **Read** | `child_status`, `difficult_life_circumstances`, `primary_registration_date` (DITY); `status_category` (CHILDWAR); факт інвалідності/потреб (ЦБІ) |
| **LRA робить** | `ssd_present/ssd_status/ssd_unaccompanied/ssd_low_income/ssd_open_month`; `childwar_status∈{deported,lost_parents,displaced}`; `has_disability` |
| **Local-only** | **W5_deportation** (`childwar_status=="deported"`, immediate) |
| **Contributes** | `ssd_*` — у W1/W6/F3/P1/F1 (контекст «вже на обліку»); `childwar_status` — у W1/W2; `has_disability` — у E4 |
| **Гейт** | Прямий онлайн-доступ ССД (`ACTORS §4.3`). Це єдина роль із інтегрованим Рівнем-2 → саме сюди сходяться чужі сигнали |

### C2 — Соціально-економічний контекст (ВПО, ПФУ-зайнятість, ЄІССС) · GATED-L2
| | |
|---|---|
| **Роль LRA** | Депривація / переміщення / нестабільність доходу — «фон ризику», майже ніколи не порушення сам по собі. |
| **Read** | `displacement_date`, `education_place`, `medical_needs` (ВПО); `employment_status_indicator` (ПФУ); виплати/пільги (ЄІССС) |
| **LRA робить** | `has_idp/idp_month/reatt_school/reatt_doctor`; `pfu_unemployed` |
| **Local-only** | — (фон, не червоний прапорець; навмисно, щоб «бідність ≠ порушення») |
| **Contributes** | `has_idp` + обрив сервісу → W1; `reatt_doctor=false` → W8; `pfu_unemployed`/`ssd_low_income` → підсилювач у F3 |
| **Гейт** | Частково прямий через ЄІССС (рольовий, `ACTORS §3.2`); решта — Трембіта |

### C3 — Освіта (ІСУО/АІКОМ, ЄДЕБО) · GATED-L2, обов'язок повідомити ССД
| | |
|---|---|
| **Роль LRA** | Розрив відвідуваності / відсів / незарахування. Тут LRA робить **часовий аналіз** (change-point), а не лише факт. |
| **Read** | `attendance_period`, `missed_lessons_count`, `score_12`, `anti_bullying_commission`, `out_of_education_flag` (АІКОМ); `study_status`, `status_effective_date`, `special_category` (ЄДЕБО) |
| **LRA робить** | `absence_spike` (change-point ≥4, ↑), `gpa_drop` (≥2, ↓), `anti_bullying`, `out_of_education`, `isuo_last_month`; `school_exit`, `enrolled_ever`, `edebo_inclusive`; вік-гейт `school_age 6–17` |
| **Local-only** | **W3_out_of_education** (вихід зі школи АБО тривала тиша АІКОМ); **F4_child_labor** (системні пропуски без булінгу/нехтування/відсіву) |
| **Contributes** | `absence_spike`+`gpa_drop` → E1; `absence_spike` → F3; `edebo_inclusive=false` → E4 |
| **Гейт** | Через заклад/орган освіти; **обов'язок повідомити ССД+поліцію на 10-й робочий день відсутності** (`ROLE_ACCESS_MATRIX §2`) — це і є push-тригер |

### C4 — Здоров'я (ЕСОЗ/eHealth) · **WALLED (de lege ferenda)**
| | |
|---|---|
| **Роль LRA** | **Спляча.** Чинної правової підстави для доступу ССД до ЕСОЗ немає навіть до булеана (`ACTORS §4.2`). LRA проєктується, але вмикається лише після нормативного запровадження. |
| **Read** (лише локально, лікарем) | `resource_type∈{declaration,encounter,condition,immunization}`, `status`, `condition_code/category`, `is_repeat` |
| **LRA робить** (потенційно) | `decl_terminated/decl_end_month`, `has_chronic`, `missed_checkup`, `trauma_present/trauma_repeat`, `psych_present` |
| **Local-only** | **W2_psych_trauma** (тільки якщо немає іншого пояснення) — але **лише як push** від лікаря, не автозапит |
| **Contributes** | `trauma_repeat` → P1; `decl_terminated`+`has_chronic` → W8; `missed_checkup` → F3/W8; `psych_present` → E1/F1/W2 |
| **Гейт** | **Заборонено pull.** Єдиний легальний вихід — **лікарське обов'язкове повідомлення про ознаки насильства (≤24 год / ≤3 год у закладі)** як push-сигнал у ССД+поліцію. Зміст медкарти — ніколи (ст. 39-1/40 Основ). У коді: emit eHealth-сигналів треба обгорнути прапорцем `legal_basis_enabled=false` |

### C5 — Кримінальна юстиція та безпека (ЄРДР, реєстр ДН, СКАЙД, ЄДРСР) · **змішаний**
| | |
|---|---|
| **Роль LRA** | Розділяється за стіною: ЄРДР — глуха стіна; ДН/СКАЙД — поліцейська вертикаль (GATED); ЄДРСР — **повністю відкритий** (LRA читає вільно). |
| **Read** | ЄРДР: `preliminary_legal_qualification`, `register_entry_datetime`; ДН: `form_of_violence`, кількість викликів; СКАЙД: факт обліку; ЄДРСР: `case_category` |
| **LRA робить** | `erdr_article∈{149,152,156,126-1}`, `erdr_month`; `police_calls`, `dv_psych`; `skaid_present`; `court_deprivation` («батьк…» у категорії справи) |
| **Local-only** | **W7_trafficking** (ЄРДР 149), **F6_sexual_abuse** (ЄРДР 152) — *immediate*, але **тільки push**; `court_deprivation` → нога W6 (ЄДРСР, вільно) |
| **Contributes** | `police_calls`/`erdr_article==126-1` → P1; `dv_psych` → F1 |
| **Гейт** | **ЄРДР = WALLED:** навіть факт/статус потерпілого = таємниця слідства (ст. 222 КПК) → лише через слідчого/прокурора з письмовим дозволом, не автосигнал. **ЄДРСР = OPEN:** позбавлення/обмеження батьк. прав, аліменти, відібрання — читаємо без запиту (ЗУ № 3262-IV), крім усиновлень (виключені) |

| | |
|---|---|
| **Роль LRA** | Нормалізує громадянські звернення в intake-сигнал; точка входу, не реєстр-джерело. |
| **Read** | `violence_type`, факт звернення |
| **LRA робить** | `hotline_present`, `hotline_psych` |
| **Contributes** | `hotline_psych` → F1 |
| **Гейт** | Канал сигналів до ССД (`ACTORS §4.1`); поліція отримує сигнал за ознаками злочину |

| | |
|---|---|
| **Роль LRA** | Відчуження житла дитини без дозволу опіки — контекст бездоглядності/порушення прав. |
| **Read** | `child_residence_alienation` |
| **LRA робить** | `housing_alienation` |
| **Local-only** | **W9_identity** (відчуження + ЄДДР) |
| **Гейт** | Відкритий/умовно-прямий (ЗУ № 1952-IV ст. 32) |

### C-EE — Естонське дзеркало (RR/RAHV, EHIS, TERVIS, SKAIS/STAR) · X-tee, крос-кордон лише PPRL
| | |
|---|---|
| **Роль LRA** | Усередині Естонії LRA працюють так само, але **зв'язуються за isikukood** через X-tee (тривіальний join). Назовні (UA↔EE) — **ніколи** isikukood. |
| **LRA робить** | Локальні естонські сигнали (RR — місце проживання/опіка; EHIS — освіта; TERVIS — медичне = WALLED як ЕСОЗ; STAR — кейс захисту) |

Жодного змісту реєстрів за кордон. Після позитивного збігу естонська сторона діє власними процедурами (`ACTORS §5`) |

## 5. Крос-реєстровий перетин (де LRA-сигнали сходяться)

LRA рахує все, **що може, локально**. Назовні йде envelope. Зведення — **тільки на агрегаторі ССД-рівня** (C1), бо лише ССД має інтегрований Рівень-2 (`ROLE_ACCESS_MATRIX §4`).

**Протокол (hit/no-hit, як Prüm/EESSI):**
1. Кожен GATED/OPEN LRA публікує `{pseudonym, indicator, осі суті}` у захищений пул (Bloom/Count-Min для дешевого членства; повний envelope — за запитом).
2. Агрегатор робить **PSI-перетин псевдонімів** (`privacy.psi_*`): шукає дитину, на яку є сигнали з **≥2 кластерів**.
3. **Гейт важливості:** перетин піднімає кейс лише якщо хоча б один сигнал — «важливий» (severity/acuity高 або immediate-порушення). Інакше — фон, кейс не створюється (так «усі різні», 77% чистих — не тонуть у шумі).
4. Формула порушення застосовується до зведених сигналів — це **рівно сьогоднішній `detect_entity`**, тільки входи прийшли з вузлів, а не зі злитого `rows_by_reg`.

**Що вирішується локально vs перетином (прямо з `detect_entity`):**

| Тип | Порушення | Кластери |
|---|---|---|
| **Local-only** (LRA сам, без перетину) | W3, F4 | C3 |
| | W5 | C1 |
| | W9 | C7 |
| | W7, F6 | C5 (але push-only, WALLED) |
| | W2 | C4 (push-only, WALLED) |
| **Cross-cluster** (потрібен PSI-перетин на ССД) | W1 | C2+C3/C4 |
| | W8 | C4+C2 |
| | W6 | C0/C5+C1 |
| | F3 | C4+C3+C1 |
| | P1 | C4+C5(+C1) |
| | E1 | C3+C4 |
| | E4 | C1+C3 |
| | F1 | C6/C5+C4/C1 |

**Маршрутизація результату** (`ROLE_ACCESS_MATRIX §2`): зведений червоний прапорець → відповідальний фахівець ССД за територією (`caseload.compute`, oblast-скоуп Ф-Г/Ф-М/Ф-О); кримінальна нога → поліція; нога ЕСОЗ/ЄРДР повертається у систему лише авторизованим push-ем.

## 6. Як це лягає на наш код (стан і що добудувати)

**Вже є (централізована LRA):**
- `detection.py:_signals` — увесь per-registry блок (це і є логіка вузлів, тільки злита).
- `detection.py:detect_entity` — формули перетину.
- `privacy.py` — `clk/dice/bloom_encode/psi_tokens/psi_membership` (примітиви для §5).
- `crossborder.py` — PPRL-міст UA↔EE (готовий C-EE).
- `caseload.py` — маршрутизація за територією.
- `access_level` на кожному реєстрі в `emitters/registries.py`.

**Щоб зробити федеративним (MVP `edge_agent.py`):**
1. Винести per-registry гілки `_signals` у функцію-вузол `lra_emit(registry, rows) -> [signal_envelope]` (один і той самий код, виклик на силос).
2. Додати `access_class` (OPEN/GATED-L2/WALLED) поряд з `access_level`; для WALLED — фізично прибрати шлях pull→emit, лишити лише `push(signal)`.
3. Агрегатор: `intersect(signals) -> entities` через `psi_membership`, потім існуючий `detect_entity`.
4. Гейт важливості (п. 5.3) перед створенням кейсу.
5. Журнал емісії (`audit_id`) — таблиця доступу.
